# Claude Code Guidelines тАФ AI Running Coach

## Git Workflow

- **Never commit directly to `master`** тАФ it is branch-protected
- Always create a feature branch before starting work: `git checkout -b feature/phase-N-name`
- When a phase is complete, open a PR against `master` using `gh pr create`
- Stop after opening the PR тАФ the user reviews and merges manually via GitHub UI
- **Always ask for confirmation before pushing** тАФ do not push on every small change; batch commits and push only when the user says so
- **ALWAYS rebase onto master before pushing тАФ NO EXCEPTIONS** тАФ run `git fetch origin && git rebase origin/master` immediately before every `git push`. This prevents merge conflicts in PRs. Skipping this causes code conflicts in the PR diff. Do it even if you think the branch is up to date.

## Architecture Decisions

- **MongoDB must be running for the app to work** тАФ if MongoDB is down, all API requests return 503. This is intentional for security. Do NOT add graceful degradation or fallback behavior when MongoDB is unavailable.
- **Azure Functions Core Tools** тАФ `func start` must be run from the `api/` directory. `azure-functions-core-tools@4` is installed as a local dev dependency in `api/` so `npm run start` always uses the correct version from `node_modules/.bin/func` regardless of system PATH. CI installs `azure-functions-core-tools-4` via apt. Do NOT use `npx func start` тАФ it resolves a different unrelated npm package named `func`.
- **`func start` requires `FUNCTIONS_WORKER_RUNTIME=node`** тАФ `local.settings.json` is gitignored, so this env var must be set explicitly in any environment that doesn't have it (CI, Playwright webServer). Without it, `func start` hangs on an interactive runtime selection prompt.
- **E2E tests use `APP_PASSWORD=e2e-test-password`** тАФ when starting the API manually to run E2E tests, set this: `APP_PASSWORD=e2e-test-password npm start` from `api/`. The Playwright webServer config sets it automatically when Playwright manages the server.
- **`ANTHROPIC_API_KEY` must never be set in test environments** тАФ `playwright.config.ts` explicitly sets `ANTHROPIC_API_KEY: ''` in the webServer env. Unit/integration tests mock `@anthropic-ai/sdk` via `vi.mock()`. Never add real API keys to test configs.
- **App commands use `<app:*/>` XML tags** тАФ Claude can emit `<app:navigate page="X"/>` and `<app:complete session_id="Y"/>` at the end of responses. `useChat.ts` strips them from the displayed message and executes them client-side after the stream closes. System prompt (`api/src/shared/prompts.ts`) defines the full protocol and includes upcoming session IDs so Claude can reference them.
- **CoachPanel mobile layout** тАФ `AppShell` owns `coachOpen` state and passes `isOpen`/`onClose` to `CoachPanel`. Mobile: `fixed inset-0 z-50` overlay. Desktop (`md+`): static right column. FAB (`fixed bottom-6 right-6 md:hidden`) in AppShell opens the panel.
- **Shared chat state via ChatContext** тАФ `useChat` is instantiated once inside `ChatProvider` (wraps AppShell in App.tsx). Both `CoachPanel` and `TrainingPlan` consume it via `useChatContext()`. Never call `useChat()` directly in components тАФ always use `useChatContext()` to avoid duplicate state instances.
- **Plan archive clears chat** тАФ `usePlan.archivePlan()` dispatches a `plan-archived` window event after success. `useChat` listens for this event and resets plan/messages state. This keeps the coach panel in sync without prop drilling.
- **Training plan goal embedded in JSON** тАФ The system prompt instructs Claude to include a `goal` object inside `<training_plan>` JSON. `generatePlan` API extracts it from the JSON (takes precedence over client-passed goal). The `objective` field is derived server-side via `deriveObjective()` to avoid client-side marathon-fallback bugs.
- **`<plan:update>` and `<plan:add>` stripped during streaming** тАФ `useChat.ts` strips both tag types from the displayed message live. After `done`, `<plan:update>` tags are applied via `PATCH /api/plan/days/:week/:day` (e.g. `week="3" day="B"`); `<plan:add>` tags are applied via `POST /api/plan/days`. Both dispatch a `plan-updated` window event so `usePlan` (TrainingPlan page) refreshes automatically without a page reload. System prompt instructs Claude to use `<plan:add>` only for new days and `<plan:update>` only for existing days.
- **Plan replace guard** тАФ `POST /api/plan/generate` and `POST /api/plan` no longer block on completed days (plan regeneration is allowed at any point). `DELETE /api/plan/days/:week/:day` returns 409 for completed days тАФ training history is locked and cannot be removed. `POST /api/plan/days` (addDay) uses `{ weekNumber, label, type, ... }` тАФ pushes a new day into the week's `days` array via `$push` (does not overwrite a rest slot). Returns 409 if the label already exists in that week, 404 if the week number is not found.
- **iOS Safari mobile fixes** тАФ `viewport-fit=cover` in `index.html` enables safe-area insets. Sidebar logout button uses `pb-[max(0.5rem,env(safe-area-inset-bottom))]` to avoid the home bar. CoachPanel uses `h-[85dvh]` (dynamic viewport height) so the panel isn't obscured by the browser chrome. Input/textarea/select font-size is forced to 16px on mobile (тЙд767px) to prevent iOS auto-zoom on focus.
- **Day operations** тАФ `PATCH /api/plan/days/:week/:day` and `DELETE /api/plan/days/:week/:day` both match plans with `status: { $in: ['active', 'onboarding'] }`. `:week` is a positive integer (global sequential week number); `:day` is AтАУG (label). PATCH supports `completed: 'false'`/`skipped: 'false'` (undo) and all field updates. The completed/skipped comparison accepts both string `'false'` and boolean `false` to guard against Azure Functions JSON runtime coercion. `POST /api/plan/days` adds a new day using `{ weekNumber, label, type, ... }` via `$push` into the target week.
- **DayRow actions** тАФ Undo and Delete buttons are always visible (no hover-to-reveal). Day label (AтАУG) is shown statically тАФ no date picker or reschedule UI. All action buttons are hidden in `readonly` mode.
- **Rest days hidden in PlanView** тАФ PlanView filters `day.type !== 'rest'` before rendering and sorts days by label alphabetically. Each week shows a `+ Add day` button (when `onAddDay` prop provided and not readonly). The `existingLabels` passed to `AddDayForm` excludes rest days тАФ rest day slots are free to overwrite with a run.
- **Empty plan archive prevention** тАФ `POST /api/plan/archive` deletes (rather than archives) plans with no workout days тАФ this covers plans with no phases AND plans with phases that contain only rest days or empty weeks. Only plans with at least one non-rest day get archived.
- **Training Plan as home** тАФ `"/"` route redirects to `"/plan"`. Dashboard moved to `"/dashboard"`. Sidebar order: Training Plan тЖТ Runs тЖТ Archive тЖТ Dashboard.
- **Chat scroll** тАФ `CoachPanel` uses a ref on the messages container div and sets `scrollTop = scrollHeight` instead of `scrollIntoView` тАФ prevents the whole page from scrolling when new messages arrive.
- **XML stripped from chat history** тАФ On mount, `useChat` strips `<training_plan>`, `<plan:update>`, `<plan:add>`, and `<app:*/>` tags from assistant messages loaded from MongoDB, since they were stored raw and must not be re-displayed.
- **Sidebar layout** тАФ Sidebar uses `h-full sticky top-0 overflow-y-auto` so it stays fixed as the main content scrolls. `AppShell` uses `h-[100dvh]` (dynamic viewport height) rather than `h-screen` (`100vh`) so Safari's bottom browser chrome does not clip the sidebar logout button.
- **System prompt тАФ no calendar** тАФ `buildSystemPrompt` in `api/src/shared/prompts.ts` no longer injects a date calendar. Plans use globally sequential week numbers (1, 2, 3...) and per-week day labels (AтАУG). Claude addresses days as `week="N" day="X"` in XML tags. When a user asks to "remove" a day, Claude must use `<plan:update week="N" day="X" skipped="true"/>` (deletion is not supported) and be transparent about it.
- **Day labels and week numbers** тАФ Days are labeled AтАУG within each week (non-rest days only; rest days have `label: ""`). Week numbers are globally sequential across all phases: phase 1 has weeks 1,2,3; phase 2 continues with 4,5,6; etc. `assignPlanStructure(phases)` in `planUtils.ts` assigns both on save.
- **plan:update and plan:add error surfacing** тАФ When `<plan:update>` PATCH or `<plan:add>` POST API calls fail (non-2xx), `useChat.ts` collects the error messages from both arrays and appends them to the assistant's chat message as `тЪая╕П <errors>` so the user sees what went wrong. This applies in both `sendMessage` and `startPlan` code paths.
- **Server-side plan saving from chat** тАФ When Claude's response includes `<training_plan>`, `chat.ts` parses and saves the plan directly in `stream.on('message')` (after `await`ing the DB write) then emits `{ done: true, planGenerated: true }`. The client never calls `POST /api/plan/generate` anymore тАФ it just re-fetches via `GET /api/plan` after receiving `planGenerated: true`. The `done` event is the synchronization barrier: by the time the client fetches, the DB write is complete. `extractGoalFromText` was removed from `useChat.ts` as it is now unused.
- **Synthetic plan-state context injection** тАФ `chat.ts` injects a synthetic `user`/`assistant` message pair immediately before the current user message when the plan has phases. The synthetic user message contains the full current plan state (all non-rest days with week number, label, status, and guidelines тАФ e.g. `Week 1 Day A [COMPLETED]: 5km тАФ Easy run`). This overrides stale references in older chat history. Not injected when the plan has no phases (onboarding).
- **+ Add day button** тАФ `PlanView.tsx` shows `+ Add day` per week when a week has fewer than 7 non-rest days (labels AтАУG are available). `hasAvailableLabel` is simply `activeDays.length < 7` тАФ no date-based computation.

## Testing & Documentation

- **`npm run build` in `web/` is mandatory before committing** тАФ Azure Static Web Apps CI runs `tsc -b && vite build` during deployment. If the TypeScript build fails, Oryx skips building and uploads the entire `web/` folder (including `node_modules`) which exceeds the 262MB SWA size limit and fails deployment. Always run `cd web && npm run build` locally to catch TypeScript errors before pushing.
- **Tests are part of execution, not UAT** тАФ creating AND running all test layers (unit, integration, E2E) is part of every phase execution, not optional cleanup. Run `npm test` in `api/` and `web/`, and `npx playwright test` for E2E. Write real test implementations that cover the phase requirements тАФ stubs are placeholders only. Do not ask the user to run tests or treat test results as human UAT items.
- **E2E tests are mandatory тАФ run `npx playwright test` after every set of changes, not just unit tests.** `e2e/global-setup.ts` starts MongoDB automatically via `docker compose up -d mongodb` тАФ Docker must be running but MongoDB does not need to be pre-started. `reuseExistingServer: true` locally means Playwright will reuse already-running API/web servers or start them via `cd api && npm start`. Fix all E2E failures before considering work done. Never skip E2E because "unit tests pass".
- **CI test count badges** тАФ CI runs vitest with `--reporter=verbose --reporter=json --outputFile.json=coverage/test-results.json` to produce a JSON results file alongside coverage. Playwright uses a conditional reporter in `playwright.config.ts`: `process.env.CI ? [['line'], ['json', { outputFile: 'playwright-results.json' }]] : 'line'`. The `coverage-badge` job (runs on `if: always()`) downloads both artifacts and updates API Tests / Web Tests / E2E Tests badges in README тАФ green if all pass, red if any fail.
- **Review and update README.md** after changes тАФ keep setup instructions, feature list, and any relevant sections accurate
- **Review and update CLAUDE.md** after changes тАФ keep architecture decisions and guidelines current
