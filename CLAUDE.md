# Claude Code Guidelines — AI Running Coach

## Git Workflow

- **Never commit directly to `master`** — it is branch-protected
- Always create a feature branch before starting work: `git checkout -b feature/phase-N-name`
- When a phase is complete, open a PR against `master` using `gh pr create`
- Stop after opening the PR — the user reviews and merges manually via GitHub UI
- **Always ask for confirmation before pushing** — do not push on every small change; batch commits and push only when the user says so
- **ALWAYS rebase onto master before pushing — NO EXCEPTIONS** — run `git fetch origin && git rebase origin/master` immediately before every `git push`. This prevents merge conflicts in PRs. Skipping this causes code conflicts in the PR diff. Do it even if you think the branch is up to date.

## Architecture Decisions

- **MongoDB must be running for the app to work** — if MongoDB is down, all API requests return 503. This is intentional for security. Do NOT add graceful degradation or fallback behavior when MongoDB is unavailable.
- **Azure Functions Core Tools** — `func start` must be run from the `api/` directory. `azure-functions-core-tools@4` is installed as a local dev dependency in `api/` so `npm run start` always uses the correct version from `node_modules/.bin/func` regardless of system PATH. CI installs `azure-functions-core-tools-4` via apt. Do NOT use `npx func start` — it resolves a different unrelated npm package named `func`.
- **`func start` requires `FUNCTIONS_WORKER_RUNTIME=node`** — `local.settings.json` is gitignored, so this env var must be set explicitly in any environment that doesn't have it (CI, Playwright webServer). Without it, `func start` hangs on an interactive runtime selection prompt.
- **E2E tests require `JWT_SECRET`** — `playwright.config.ts` webServer env must include `JWT_SECRET`. The global setup (`e2e/global-setup.ts`) seeds test users into MongoDB so E2E tests can authenticate. Tests obtain JWT access tokens by calling `POST /api/auth/login` directly; tokens are stored in test fixtures and injected into `localStorage` via `page.addInitScript`.
- **`ANTHROPIC_API_KEY` must never be set in test environments** — `playwright.config.ts` explicitly sets `ANTHROPIC_API_KEY: ''` in the webServer env. Unit/integration tests mock `@anthropic-ai/sdk` via `vi.mock()`. Never add real API keys to test configs.
- **Auth middleware pattern** — Every route handler calls `requireAuth(req)` first (returns 401 if invalid), then immediately calls `getAuthContext(req)` to extract `{ userId, email, isAdmin }` from the WeakMap populated by `requireAuth`. Never call `getAuthContext` before `requireAuth` — it throws. Use `requireAdmin(req)` for admin-only routes (calls `requireAuth` internally). All three are exported from `api/src/middleware/auth.ts`.
- **Admin API routes use `/api/users` prefix, NOT `/api/admin/users`** — Azure Functions Core Tools reserves `/admin` as the host management API path. Any Azure Functions HTTP handler with `route: 'admin/...'` will be shadowed and return 404. Admin handlers in `api/src/functions/admin.ts` use `route: 'users'` and `route: 'users/{id}/...'` instead, accessed at `/api/users` and `/api/users/:id/...`. They are protected by `requireAdmin()` so non-admin requests receive 401.
- **`LoginPage` decodes JWT to extract `isAdmin`** — The login API response does NOT include `isAdmin` in the JSON body (only `token`, `refreshToken`, `expiresIn`, `tempPassword`). `LoginPage.tsx` decodes the JWT payload with `atob(token.split('.')[1])` to extract `isAdmin` and passes it to `login()`. This avoids adding `isAdmin` to the login response body while still setting the admin flag correctly in React state.
- **E2E test user isolation** — `e2e/global-setup.ts` seeds 5 users: `test@example.com` (normal), `temp@example.com` (temp password), `userb@example.com` (cross-user isolation tests), `admin@example.com` (admin panel tests), `deactivate@example.com` (deactivation/login-rejection tests). Do NOT use `test@example.com` or `userb@example.com` in destructive E2E tests (password reset, deactivation) — they are shared by other specs. Use `deactivate@example.com` for those tests.
- **Startup migration — `index.ts` call order matters** — `api/src/index.ts` calls `runStartupMigration()` (from `api/src/shared/migration.ts`) before all route imports. This backfills pre-v2.0 documents (no `userId`) to the seed admin user on first deployment. The call is non-blocking (`.catch()` only) so a migration failure never prevents the API from starting. Do NOT move this call after route imports — it must run before any request can be served.
- **AuthContext and token storage** — `AuthContext` (`web/src/contexts/AuthContext.tsx`) stores `token`, `email`, `isAdmin`, `tempPassword` in React state, seeded from `localStorage` on mount. `login()` writes five keys: `access_token`, `refresh_token`, `auth_email`, `auth_is_admin`, `auth_temp_password`. `logout()` removes all five. Use `useAuth()` to access auth state — never read `localStorage` directly in components.
- **App-level auth gate** — `App.tsx` wraps everything in `AuthProvider`. The inner component `AppInner` gates rendering: if `!token` → render `<LoginPage>` (no BrowserRouter, no routes); if `tempPassword` → render `<ChangePasswordPage>` (ditto); otherwise render the full app with `BrowserRouter`, `ChatProvider`, and `AppShell`. This means neither `LoginPage` nor `ChangePasswordPage` are reachable via URL — they are conditionally rendered states, not routes.
- **401 interceptor — global fetch override** — `AppInner` installs a `window.fetch` override via `useEffect` that intercepts every 401 response. Auth endpoints (`/api/auth/login`, `/api/auth/refresh`, `/api/auth/logout`) are skipped to prevent infinite loops. On 401, it calls `POST /api/auth/refresh` with the stored refresh token; on success, updates `localStorage` and retries the original request with the new token. Concurrent 401s during a refresh are queued and drained once the single refresh resolves. On refresh failure, `logout()` is called, which clears localStorage and sets `token: null`, causing `AppInner` to render `<LoginPage>`.
- **ChangePasswordPage clears tempPassword** — before calling `POST /api/auth/change-password`, the page proactively calls `POST /api/auth/refresh` to get a fresh access token (stored in `localStorage.access_token`). This avoids relying on the 401 interceptor for the refresh, which is unreliable during this specific flow. The change-password request is then sent with the refreshed token. On success, calls `login(freshToken, refreshToken, email, isAdmin, false)` to clear the `tempPassword` flag. On 401 response, calls `logout()` directly. `/api/auth/change-password` is in the 401 interceptor skip list (`App.tsx`) so the interceptor never double-handles it.
- **App commands use `<app:*/>` XML tags** — Claude can emit `<app:navigate page="X"/>` and `<app:complete session_id="Y"/>` at the end of responses. `useChat.ts` strips them from the displayed message and executes them client-side after the stream closes. System prompt (`api/src/shared/prompts.ts`) defines the full protocol and includes upcoming session IDs so Claude can reference them. Plan-mutating tags (`<plan:update>`, `<plan:add>`, `<plan:unlink>`) are also stripped during streaming and processed in `applyPlanOperations`. `<plan:unlink week="N" day="X"/>` unlinks the run linked to that plan day.
- **Plan badge color** — The Week/Day badge on a run (in `RunDetailModal` and the Runs list) is green when `run.planId === activePlan._id` (run belongs to the current active plan) and gray when the plan has been archived. `activePlanId` is passed as a prop from the page-level `usePlan()` call; never call `usePlan()` inside `RunDetailModal` or `RunRow`.
- **CoachPanel mobile layout** — `AppShell` owns `coachOpen` state and passes `isOpen`/`onClose` to `CoachPanel`. Mobile: `fixed inset-0 z-50` overlay. Desktop (`md+`): static right column. FAB (`fixed bottom-6 right-6 md:hidden`) in AppShell opens the panel.
- **CoachPanel readonly mode** — `CoachPanel` accepts `readonly?: boolean` and `initialMessages?: {role, content}[]` props. When `readonly={true}`: uses `initialMessages` instead of `useChatContext()` messages (hook still called unconditionally per React rules), shows "Plan History" as title in `text-gray-500`, hides the textarea input, Send button, Start Over button, and error banner. XML tags are stripped from `initialMessages` before rendering. Used by `ArchivePlan` to show archived chat history — `ArchivePlan` fetches `/api/messages?planId=X` on mount and passes results as `initialMessages`. The FAB on `ArchivePlan` uses `bg-gray-500` (not blue) to signal readonly/history mode.
- **ArchivePlan uses PlanView (not ReactMarkdown)** — `ArchivePlan` renders the archived plan using `<PlanView readonly={true} linkedRuns={linkedRuns} />`, the same component used by `TrainingPlan`. `GET /api/plans/archived/:id` returns `{ plan, linkedRuns }` (same shape as `GET /api/plan`). `ArchivePlan` converts `linkedRuns` from `Record<string, Run>` to `Map<string, Run>` before passing to `PlanView`. Clicking a linked run date dispatches an `open-run-detail` event (same as `TrainingPlan`) and `ArchivePlan` listens for it, fetches the run via `GET /api/runs/:id`, and opens `RunDetailModal`. `planToMarkdown` utility is unused and deleted — do not re-add it.
- **Shared chat state via ChatContext** — `useChat` is instantiated once inside `ChatProvider` (wraps AppShell in App.tsx). Both `CoachPanel` and `TrainingPlan` consume it via `useChatContext()`. Never call `useChat()` directly in components — always use `useChatContext()` to avoid duplicate state instances.
- **Plan archive clears chat** — `usePlan.archivePlan()` dispatches a `plan-archived` window event after success. `useChat` listens for this event and resets plan/messages state. This keeps the coach panel in sync without prop drilling.
- **Training plan goal embedded in JSON** — The system prompt instructs Claude to include a `goal` object inside `<training_plan>` JSON. `generatePlan` API extracts it from the JSON (takes precedence over client-passed goal). The `objective` field is derived server-side via `deriveObjective()` to avoid client-side marathon-fallback bugs.
- **`<plan:update>` and `<plan:add>` stripped during streaming** — `useChat.ts` strips both tag types from the displayed message live. After `done`, `<plan:update>` tags are applied via `PATCH /api/plan/days/:week/:day` (e.g. `week="3" day="B"`); `<plan:add>` tags are applied via `POST /api/plan/days`. Both dispatch a `plan-updated` window event so `usePlan` (TrainingPlan page) refreshes automatically without a page reload. System prompt instructs Claude to use `<plan:add>` only for new days and `<plan:update>` only for existing days.
- **Phase 5 agent commands** — Six additional XML tags handled by `useChat.ts` in `applyPlanOperations`: `<plan:add-phase name="X" description="Y"/>` (POST `/api/plan/phases`), `<plan:add-week phaseIndex="N"/>` (POST `/api/plan/phases/N/weeks`), `<plan:update-goal targetDate="YYYY-MM-DD"/>` (PATCH `/api/plan` with `targetDate`; empty string clears it), `<plan:update-feedback feedback="..."/>` (PATCH `/api/plan` with `progressFeedback`; skipped if `feedback` attr is empty), `<run:create date="..." distance="N" unit="km|miles" duration="..."/>` (POST `/api/runs`; `unit` attr intentionally NOT forwarded), `<run:update-insight runId="..." insight="..."/>` (PATCH `/api/runs/:runId`). All are stripped from the displayed message during streaming. `run:create` auto-saves the full response text as the run's `insight` immediately after POST (since no RunId exists when the stream starts). `plan:update-feedback` and `run:update-insight` do NOT dispatch `plan-updated` — they are insight-only with no structural plan change.
- **`POST /api/plan/phases`** — Adds a new phase to the active plan. Body: `{ name?, description? }`. Uses `assignPlanStructure` to recompute globally sequential week numbers across all phases. Returns 201 with the updated plan. Auto-names the phase `Phase N` if no name provided. New phase starts with one empty week (no days).
- **`POST /api/plan/phases/:phaseIndex/weeks`** — Adds an empty week to the specified phase (0-based index). `:phaseIndex` must be a non-negative integer; returns 400 if invalid, 404 if out of bounds. Uses `assignPlanStructure` to recompute week numbers globally after insertion. Returns 201 with the updated plan.
- **`PATCH /api/plan` `targetDate` field** — `PATCH /api/plan` now accepts `targetDate` in addition to `progressFeedback`. A non-empty string sets the field; an empty string `""` issues a MongoDB `$unset` to remove it. Returns 400 only when neither `progressFeedback` nor `targetDate` is provided.
- **Inline target date editor** — The TrainingPlan header shows the plan's `targetDate` as a clickable span ("Target: YYYY-MM-DD"). Clicking enters edit mode (renders a `<input type="date">`). Enter/blur saves via `PATCH /api/plan`. Escape reverts without saving. When `targetDate` is absent, shows a "+ Set target date" placeholder that also enters edit mode on click. Syncs with plan state changes (e.g. from agent `<plan:update-goal>` tags) via `useEffect` on `plan?.targetDate`.
- **`+ Add phase` / `+ Add week` buttons** — `PlanView` renders a `+ Add week` button below each phase's week list (when `onAddWeek` prop is provided and not readonly) and a `+ Add phase` button at the bottom of all phases (when `onAddPhase` prop is provided and not readonly). Both call `usePlan.addPhase()` / `usePlan.addWeek(phaseIndex)` respectively, which POST to the API and re-fetch the plan.
- **RunId in synthetic plan-state context** — `chat.ts` appends `| RunId: <id>` to each completed day line in the synthetic plan-state context. This allows Claude to reference run IDs when emitting `<run:update-insight runId="..."/>` tags for post-run coaching feedback.
- **Plan replace guard** — `POST /api/plan/generate` blocks with 409 if any linked runs exist (`planId` field present on any run document) — training history cannot be destroyed by regenerating the plan. `POST /api/plan` no longer blocks on completed days. `DELETE /api/plan/days/:week/:day` returns 409 for completed days — training history is locked. `POST /api/plan/days` (addDay) uses `{ weekNumber, label, type, ... }` — pushes a new day into the week's `days` array via `$push` (does not overwrite a rest slot). Returns 409 if the label already exists in that week, 404 if the week number is not found.
- **Run data model** — Runs are stored in a `runs` MongoDB collection. Schema: `date` (YYYY-MM-DD string, user-entered), `distance` (number), `duration` (MM:SS or HH:MM:SS string), `pace` (computed decimal min/km), `avgHR?`, `notes?`, `planId?` (ObjectId — set when linked to a plan day), `weekNumber?`, `dayLabel?`, `insight?` (coaching text saved after feedback). Routes: `POST /api/runs`, `GET /api/runs` (paginated, filters: dateFrom/dateTo/minDistance/maxDistance/planId/unlinked), `GET /api/runs/{id}`, `PATCH /api/runs/{id}`, `DELETE /api/runs/{id}` (409 if linked), `POST /api/runs/{id}/link` (retroactively links to a plan day), `POST /api/runs/{id}/unlink` (removes plan link, reverts day to incomplete). **`insight` must be saved from the `sendMessage()` return value** — never read from the `messages` state array, which is stale at the time `updateRun` runs (the state update batches after the await).
- **Run-plan linking** — `POST /api/runs` accepts optional `weekNumber`+`dayLabel` to link at create time, atomically marking the plan day as completed. `POST /api/runs/{id}/link` retroactively links; returns 409 only if the day is completed AND already has a linked run (allows linking to a day completed manually). Undo (`PATCH /api/plan/days/:week/:day` with `completed: false`) clears `planId/weekNumber/dayLabel` from the linked run but does not delete it. `POST /api/runs/{id}/unlink` is the explicit unlink: clears plan fields from the run and reverts the plan day to `completed: false` via arrayFilters. The coach can also unlink via `<plan:unlink week="N" day="X"/>` — `useChat.ts` strips the tag during streaming and processes it in `applyPlanOperations` (fetches plan → finds linked run → calls unlink API).
- **`PATCH /api/plan`** — Saves `progressFeedback` (string) to the active plan. Used by `TrainingPlan.handleGetFeedback` after the coach responds to a "how am I doing?" prompt. Returns 400 if no updatable fields provided, 404 if no active plan. **`progressFeedback` must be XML-stripped before saving** — use `responseText.replace(/<[^>]+\/>/g, '').trim()` because Claude responses may contain self-closing XML tags (`<plan:update .../>`, `<app:navigate .../>`) that must not be stored in the DB.
- **Run context in synthetic plan state** — `chat.ts` enriches the synthetic plan-state context for completed days: fetches all runs linked to the plan via `planId`, builds a `Map<"weekNumber-dayLabel", Run>` for O(1) lookup, and appends ` | Ran: DD/MM/YYYY, Xkm @ M:SS/km | Notes: <truncated 500 chars> | Insight: <truncated 150 chars>` to each completed day line. The run fetch is wrapped in a non-fatal try/catch so a missing `runs` collection never blocks the chat flow.
- **`GET /api/plan` returns `linkedRuns`** — The plan endpoint returns `{ plan, linkedRuns: Record<string, Run> }` where `linkedRuns` is keyed by `"weekNumber-dayLabel"` (e.g. `"1-A"`). `usePlan.ts` converts it to a `Map<string, Run>` and exposes it as `linkedRuns`. `PlanView` receives `linkedRuns` as a prop — it does NOT fetch runs separately. When a run is linked/unlinked, `PlanView` dispatches `plan-updated` so `usePlan` re-fetches (which re-fetches `linkedRuns` too). Never add a separate `/api/runs?planId=...` fetch inside `PlanView`.
- **Confirmation dialogs use `window.confirm`** — All destructive actions (delete run, delete training day, unlink run, delete phase, archive plan) use `window.confirm()` for consistency. Do not use inline two-step confirmation UI (e.g., "Are you sure? Yes/No" buttons rendered inside the component).
- **`cursor-pointer` enforcement** — `web/src/index.css` has a global rule `button, a, [role="button"] { cursor: pointer; }`. All interactive elements automatically get cursor-pointer. When adding new interactive elements (custom divs, spans), explicitly add `cursor-pointer` Tailwind class.
- **iOS Safari mobile fixes** — `viewport-fit=cover` in `index.html` enables safe-area insets. `body` has `background-color: #111827` (gray-900) in `index.css` so the bottom safe area zone blends with the sidebar instead of showing a white rectangle; `#root` has `background-color: #f9fafb` (gray-50) to match the main content area. Sidebar logout button uses `pb-[max(0.5rem,env(safe-area-inset-bottom))]` to avoid the home bar. CoachPanel uses `h-[85dvh]` (dynamic viewport height) so the panel isn't obscured by the browser chrome. Input/textarea/select font-size is forced to 16px on mobile (≤767px) to prevent iOS auto-zoom on focus. Date inputs use `type="date"` (native picker) with `min="2000-01-01"` and JS `isValidDate()` validation — `RunEntryForm` also sets `max` to today to prevent logging future runs.
- **Day operations** — `PATCH /api/plan/days/:week/:day` and `DELETE /api/plan/days/:week/:day` both match plans with `status: { $in: ['active', 'onboarding'] }`. `:week` is a positive integer (global sequential week number); `:day` is A–G (label). PATCH supports `completed: 'false'`/`skipped: 'false'` (undo) and all field updates. The completed/skipped comparison accepts both string `'false'` and boolean `false` to guard against Azure Functions JSON runtime coercion. `POST /api/plan/days` adds a new day using `{ weekNumber, label, type, ... }` via `$push` into the target week.
- **DayRow actions** — Undo and Delete buttons are always visible (no hover-to-reveal). Day label (A–G) is shown statically — no date picker or reschedule UI. All action buttons are hidden in `readonly` mode.
- **Rest days hidden in PlanView** — PlanView filters `day.type !== 'rest'` before rendering and sorts days by label alphabetically. Each week shows a `+ Add day` button (when `onAddDay` prop provided and not readonly). The `existingLabels` passed to `AddDayForm` excludes rest days — rest day slots are free to overwrite with a run.
- **Empty plan archive prevention** — `POST /api/plan/archive` deletes (rather than archives) plans with no workout days — this covers plans with no phases AND plans with phases that contain only rest days or empty weeks. Only plans with at least one non-rest day get archived.
- **Dashboard as home** — `"/"` route redirects to `"/dashboard"`. Training Plan at `"/plan"`. Sidebar order: Dashboard → Training Plan → Runs → Archive.
- **Chat scroll** — `CoachPanel` uses a ref on the messages container div and sets `scrollTop = scrollHeight` instead of `scrollIntoView` — prevents the whole page from scrolling when new messages arrive.
- **XML stripped from chat history** — On mount, `useChat` strips `<training_plan>`, `<plan:update>`, `<plan:add>`, and `<app:*/>` tags from assistant messages loaded from MongoDB, since they were stored raw and must not be re-displayed.
- **Sidebar layout** — Sidebar uses `h-full sticky top-0 overflow-y-auto` so it stays fixed as the main content scrolls. `AppShell` uses `h-[100dvh]` (dynamic viewport height) rather than `h-screen` (`100vh`) so Safari's bottom browser chrome does not clip the sidebar logout button.
- **System prompt — no calendar** — `buildSystemPrompt` in `api/src/shared/prompts.ts` no longer injects a date calendar. Plans use globally sequential week numbers (1, 2, 3...) and per-week day labels (A–G). Claude addresses days as `week="N" day="X"` in XML tags. When a user asks to "remove" a day, Claude must use `<plan:update week="N" day="X" skipped="true"/>` (deletion is not supported) and be transparent about it.
- **Day labels and week numbers** — Days are labeled A–G within each week (non-rest days only; rest days have `label: ""`). Week numbers are globally sequential across all phases: phase 1 has weeks 1,2,3; phase 2 continues with 4,5,6; etc. `assignPlanStructure(phases)` in `planUtils.ts` assigns both on save.
- **plan:update and plan:add error surfacing** — When `<plan:update>` PATCH or `<plan:add>` POST API calls fail (non-2xx), `useChat.ts` collects the error messages from both arrays and appends them to the assistant's chat message as `⚠️ <errors>` so the user sees what went wrong. This applies in both `sendMessage` and `startPlan` code paths.
- **Server-side plan saving from chat** — When Claude's response includes `<training_plan>`, `chat.ts` parses and saves the plan directly in `stream.on('message')` (after `await`ing the DB write) then emits `{ done: true, planGenerated: true }`. The client never calls `POST /api/plan/generate` anymore — it just re-fetches via `GET /api/plan` after receiving `planGenerated: true`. The `done` event is the synchronization barrier: by the time the client fetches, the DB write is complete. `extractGoalFromText` was removed from `useChat.ts` as it is now unused.
- **Synthetic plan-state context injection** — `chat.ts` injects a synthetic `user`/`assistant` message pair immediately before the current user message when the plan has phases. The synthetic user message contains the full current plan state (all non-rest days with week number, label, status, and guidelines — e.g. `Week 1 Day A [COMPLETED]: 5km — Easy run`). This overrides stale references in older chat history. Not injected when the plan has no phases (onboarding).
- **+ Add day button** — `PlanView.tsx` shows `+ Add day` per week when a week has fewer than 7 non-rest days (labels A–G are available). `hasAvailableLabel` is simply `activeDays.length < 7` — no date-based computation.
- **Phase edit/delete** — `PATCH /api/plan/phases/:phaseIndex` updates phase name/description (0-based index). `DELETE /api/plan/phases/last` removes the last phase (returns 400 if only one phase, 409 if completed days exist). Agent commands: `<plan:update-phase index="0" name="X" description="Y"/>` and `<plan:delete-phase/>`. `useChat.ts` strips both tag types during streaming. `PhaseHeader.tsx` provides inline editing (click-to-edit title/description) and a delete button visible only on the last phase when `totalPhases > 1`.

## Testing & Documentation

- **`npm run build` in `web/` is mandatory before committing** — Azure Static Web Apps CI runs `tsc -b && vite build` during deployment. If the TypeScript build fails, Oryx skips building and uploads the entire `web/` folder (including `node_modules`) which exceeds the 262MB SWA size limit and fails deployment. Always run `cd web && npm run build` locally to catch TypeScript errors before pushing.
- **Tests are part of execution, not UAT** — creating AND running all test layers (unit, integration, E2E) is part of every phase execution, not optional cleanup. Run `npm test` in `api/` and `web/`, and `npx playwright test` for E2E. Write real test implementations that cover the phase requirements — stubs are placeholders only. Do not ask the user to run tests or treat test results as human UAT items.
- **E2E tests are mandatory — run `npx playwright test` after every set of changes, not just unit tests.** `e2e/global-setup.ts` starts MongoDB automatically via `docker compose up -d mongodb` — Docker must be running but MongoDB does not need to be pre-started. `reuseExistingServer: true` locally means Playwright will reuse already-running API/web servers or start them via `cd api && npm start`. Fix all E2E failures before considering work done. Never skip E2E because "unit tests pass".
- **CI test count badges** — CI runs vitest with `--reporter=verbose --reporter=json --outputFile.json=coverage/test-results.json` to produce a JSON results file alongside coverage. Playwright uses a conditional reporter in `playwright.config.ts`: `process.env.CI ? [['line'], ['json', { outputFile: 'playwright-results.json' }]] : 'line'`. The `coverage-badge` job (runs on `if: always()`) downloads both artifacts and updates API Tests / Web Tests / E2E Tests badges in README — green if all pass, red if any fail.
- **All interactive elements must have cursor-pointer** — `button`, `a`, and `[role="button"]` elements get `cursor: pointer` automatically via a global rule in `web/src/index.css`. When adding custom clickable elements (e.g. `div` with `onClick`), add `cursor-pointer` Tailwind class explicitly.
- **Review and update README.md** after changes — keep setup instructions, feature list, and any relevant sections accurate
- **Review and update CLAUDE.md** after changes — keep architecture decisions and guidelines current

## GSD Workflow

- This project uses [Get Shit Done (GSD)](https://github.com/gsd-build/get-shit-done)
- Planning docs live in `.planning/`
- Phase sequence: plan → execute → PR → (user merges) → next phase
- Run `/gsd:plan-phase N` to plan a phase, `/gsd:execute-phase N` to build it
