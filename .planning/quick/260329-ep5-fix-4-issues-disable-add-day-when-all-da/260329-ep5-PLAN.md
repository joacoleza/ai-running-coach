# Quick Task 260329-ep5: 4 Quick Fixes

## Tasks

### Task 1: Disable + Add day for fully-past weeks

**Files:** `web/src/components/plan/PlanView.tsx`

Compute `hasAvailableDay` per week: a week gets the + Add day button only if at least one slot is both today-or-future AND not already occupied. Weeks where all 7 days are in the past never show the button.

**Tests:** `web/src/__tests__/PlanView.test.tsx` — "hides + Add day button when all 7 days of the week are in the past"

### Task 2: Synthetic plan-state context injection

**Files:** `api/src/functions/chat.ts`

Inject a synthetic user/assistant message pair immediately before the current user message when the plan has phases. The synthetic user message contains all non-rest days with date, status, and guidelines. Overrides stale references in older chat history.

**Tests:** `api/src/__tests__/chat.integration.test.ts` — "injects synthetic plan-state pair before the user message when plan has phases" / "does NOT inject synthetic pair when plan has no phases"

### Task 3: Fix startDate normalization

**Files:** `api/src/shared/planUtils.ts`

Ensured `normalizeWeekDays` correctly sorts and normalizes week day entries when parsing plans from Claude responses.

**Tests:** `api/src/__tests__/planUtils.test.ts`

### Task 4: Server-side plan saving from chat

**Files:** `api/src/functions/chat.ts`, `web/src/hooks/useChat.ts`, `web/src/__tests__/useChat.trainingPlan.test.ts`

When Claude's response includes `<training_plan>`, `chat.ts` parses and saves the plan directly in `stream.on('message')` (after awaiting the DB write) then emits `{ done: true, planGenerated: true }`. Client never calls `POST /api/plan/generate` — it re-fetches via `GET /api/plan` after receiving `planGenerated: true`. The `done` event is the synchronization barrier.

**Tests:** `api/src/__tests__/chat.integration.test.ts` — "saves plan to DB and emits planGenerated=true" / `web/src/__tests__/useChat.trainingPlan.test.ts` — "does NOT call /api/plan/generate when server signals planGenerated"
