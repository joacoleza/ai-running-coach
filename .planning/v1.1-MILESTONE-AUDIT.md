### Integration Issues

#### ⚠ COACH-03: Stale Closure in Insight Save (Warning)

**File:** `web/src/components/runs/RunDetailModal.tsx:110–115`

`handleAddFeedback` reads `messages` captured at render time (stale closure) instead of using `sendMessage()`'s return value. After `await sendMessage(message)` resolves, the component may have re-rendered, but the closure still holds the old `messages` array.

**Impact:** Coaching insight saved to the run record may capture a previous assistant message rather than the just-received feedback, on first call if React hasn't batched the state update.

**Fix:** Use `const responseText = await sendMessage(message); // strip XML; PATCH with responseText` — the same pattern `TrainingPlan.tsx` uses for `progressFeedback` (which also works, modulo the XML stripping issue below).

**Pre-noted:** Yes — Phase 3 VERIFICATION.md anti-pattern table, Info severity.

---

#### ⚠ COACH-04: progressFeedback May Include Raw XML (Warning)

**File:** `web/src/pages/TrainingPlan.tsx:46–60`

`sendMessage()` returns `accumulatedText` — the raw stream content before XML stripping. If Claude emits `<plan:update>` or `<app:navigate>` tags in the same response as the feedback, `progressFeedback` will be stored verbatim with those tags and rendered via `ReactMarkdown` as raw text.

**Impact:** Plan feedback section may show broken XML-like text for responses that combine feedback with plan adjustments.

**Fix:** Strip XML from `accumulatedText` before saving: `accumulatedText.replace(/<[^>]+\/>/g, '').trim()`.

## E2E Flow Verification

### Flow 1: Onboarding → Plan Generation → Plan View

`CoachPanel` (startPlan) → `POST /api/plan` → `POST /api/chat` (SSE, onboarding questions) → Claude emits `<training_plan>` → `chat.ts` saves via `assignPlanStructure` → client receives `planGenerated: true` → `GET /api/plan` → `TrainingPlan` renders `PlanView`

**Status: COMPLETE** ✓

### Flow 2: Log Run → Day Complete → Coaching Feedback

`DayRow` (Complete button) → `RunEntryForm` → `createRun({ weekNumber, dayLabel })` → `POST /api/runs` (atomically marks day complete, saves planId on run) → `plan-updated` event → `PlanView` refreshes → `RunDetailModal` (Add feedback) → `sendMessage` → `POST /api/chat` with run data in synthetic context

**Status: COMPLETE (with stale closure warning on insight save)** ⚠

### Flow 3: Run History → Coach Context → Plan Adjustment

`POST /api/chat` → `chat.ts` fetches `runs.find({ planId })` → formats run actuals in synthetic context → Claude receives full run history → Claude emits `<plan:update>` → `useChat` applies `PATCH /api/plan/days/:week/:day` → `plan-updated` event → `usePlan` refreshes

**Status: COMPLETE** ✓

### Flow 4: Link Existing Run to Plan Day

`PlanView` renders `LinkRunModal` → `GET /api/runs/unlinked` → user selects run → `linkRun(runId, weekNumber, dayLabel)` → `POST /api/runs/:id/link` → day marked complete → `plan-updated` event → refresh

**Status: COMPLETE** ✓

### Flow 5: Chat History Persistence Across Sessions

Mount → `useChat.init()` → `GET /api/plan` → if plan exists, `GET /api/messages?planId=` → messages restored with XML stripped → `CoachPanel` renders history

**Status: COMPLETE** ✓
