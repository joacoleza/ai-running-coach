## Summary

Phase 1.2 establishes a comprehensive testing foundation across three layers: (1) unit tests for API middleware and React components, (2) MongoDB integration tests with in-memory server, and (3) Playwright E2E tests against the full local stack. All three layers run in GitHub Actions on every PR.

The most important finding from direct code verification: **`HttpRequest` and `InvocationContext` are real constructable classes in `@azure/functions` v4**. No stub libraries are needed. `new HttpRequest({ method, url, headers })` and `new InvocationContext({ functionName, invocationId })` work out of the box — this is by design in v4, which was explicitly built to make unit testing easier than v3.

The strategy separates fast unit tests (no I/O, mocked MongoDB) from slower integration tests (real in-process MongoDB via mongodb-memory-server). E2E tests run against a started local stack using Playwright's `webServer` option. Coverage is tracked only for web (Vitest has native coverage support) and API (separate Vitest project in Node environment); combined badge is generated from web coverage.

**Primary recommendation:** Use three separate test scopes — `api/` uses its own Vitest config in Node environment, `web/` extends the existing jsdom Vitest config, and `e2e/` uses Playwright with webServer. One CI workflow runs all three plus coverage badge generation.

### Pattern 1: Azure Functions v4 Unit Test (VERIFIED)

**What:** Test exported handler functions directly, constructing real `HttpRequest` and `InvocationContext` objects without any stub library.

**When to use:** All API function and middleware unit tests.

**Example (verified against installed `@azure/functions` 4.11.2):**
```typescript
// api/src/__tests__/auth.test.ts
import { describe, it, expect, vi, beforeEach } from 'vitest'
import { HttpRequest, InvocationContext } from '@azure/functions'

// Mock MongoDB module-level singleton
vi.mock('mongodb', () => ({
  MongoClient: vi.fn().mockImplementation(() => ({
    connect: vi.fn().mockResolvedValue(undefined),
    db: vi.fn().mockReturnValue({
      collection: vi.fn().mockReturnValue({
        findOne: vi.fn(),
        findOneAndUpdate: vi.fn(),
        updateOne: vi.fn(),
      }),
    }),
  })),
}))

import { requirePassword } from '../middleware/auth.js'

function makeRequest(password?: string) {
  return new HttpRequest({
    method: 'GET',
    url: 'http://localhost/api/test',
    headers: password ? { 'x-app-password': password } : {},
  })
}

const mockContext = new InvocationContext({
  functionName: 'test',
  invocationId: 'test-invocation',
})

describe('requirePassword', () => {
  beforeEach(() => {
    process.env.APP_PASSWORD = 'secret'
    vi.clearAllMocks()
  })

  it('returns null when password is correct', async () => {
    // configure mocks: not blocked, password matches
    const result = await requirePassword(makeRequest('secret'))
    expect(result).toBeNull()
  })

  it('returns 401 when password is wrong', async () => {
    const result = await requirePassword(makeRequest('wrong'))
    expect(result?.status).toBe(401)
  })

  it('returns 503 when site is blocked', async () => {
    // configure findOne mock to return { blocked: true }
    const result = await requirePassword(makeRequest('anything'))
    expect(result?.status).toBe(503)
  })
})
```

### Pattern 2: MongoDB Integration Test (Real In-Memory DB)

**What:** Use `mongodb-memory-server` to spin up a real MongoDB process for tests that assert on DB state — not just return values.

**When to use:** Lockout algorithm tests where you need to verify `failureCount` and `blocked` fields in the `auth` collection after operations.

**Example:**
```typescript
// api/src/__tests__/lockout.integration.test.ts
import { MongoMemoryServer } from 'mongodb-memory-server'
import { MongoClient } from 'mongodb'
import { describe, it, expect, beforeAll, afterAll, beforeEach } from 'vitest'

let mongod: MongoMemoryServer
let uri: string

beforeAll(async () => {
  mongod = await MongoMemoryServer.create()
  uri = mongod.getUri()
  process.env.MONGODB_CONNECTION_STRING = uri
  process.env.APP_PASSWORD = 'correct-password'
})

afterAll(async () => {
  await mongod.stop()
})

beforeEach(async () => {
  // Reset auth collection between tests
  const client = new MongoClient(uri)
  await client.connect()
  await client.db('running-coach').collection('auth').deleteMany({})
  await client.close()
})

it('increments failureCount on wrong password', async () => {
  const { requirePassword } = await import('../middleware/auth.js')
  // ... call requirePassword with wrong password
  // ... connect directly to mongod and assert failureCount === 1
})

it('sets blocked=true after 30 failures', async () => {
  // call requirePassword 30 times with wrong password
  // assert blocked === true in auth collection
})
```

**Important:** When using `mongodb-memory-server` for module-level singleton tests (like `auth.ts` which caches the MongoClient), tests must reset the module between tests or isolate the connection string before import. Use Vitest's `vi.resetModules()` in `beforeEach` if needed.

### Anti-Patterns to Avoid

- **Testing `app.http(...)` registration instead of the handler:** The `app.http()` call is SDK wiring, not business logic. Export the handler function separately and test it directly.
- **Global MongoClient singleton across tests without reset:** The `auth.ts` module caches a MongoDB connection at module scope. Tests that import it will share state. Use `vi.resetModules()` and `vi.unstubAllEnvs()` between integration tests.
- **Running E2E against `npm run dev`:** `npm run dev` starts Docker Compose which starts all services including the Cosmos DB emulator. In CI this is too heavy. Start only MongoDB (via Action) and the Functions/Vite servers directly.
- **Using jest-mongodb or shelfio/jest-mongodb:** These are Jest-specific presets. For Vitest, use `mongodb-memory-server` directly with `beforeAll/afterAll` hooks.
- **jsdom environment for API tests:** Azure Functions run in Node. `jsdom` is for browser-like environments. API vitest config must use `environment: 'node'`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| In-memory MongoDB | Custom fake MongoDB | `mongodb-memory-server` | Handles binary download, port allocation, start/stop lifecycle |
| HttpRequest mocking | Manual object with type cast | `new HttpRequest(init)` from `@azure/functions` | Official class, all headers/methods work correctly |
| InvocationContext mocking | `{} as InvocationContext` | `new InvocationContext({...})` | Official class, v4 designed for this |
| Coverage badges | Custom shell script | `jpb06/coverage-badges-action` | Handles SVG generation, git commit, branch targeting |
| Browser install in CI | Custom install scripts | `npx playwright install --with-deps` | Installs OS-level browser dependencies automatically |
| CI MongoDB service | Docker Compose in CI | `supercharge/mongodb-github-action` | Single step, version-pinned, health-checked |

**Key insight:** Azure Functions v4 was explicitly redesigned to make unit testing possible without special test utilities. The v4 migration docs include a dedicated "Create a test context" section. Embrace this — the pattern is `new HttpRequest(init)` + `new InvocationContext(init)`, period.

## Common Pitfalls

### Pitfall 1: Module-Level Singleton Breaks Between Tests

**What goes wrong:** `auth.ts` caches `client` and `db` at module scope. If two tests import `requirePassword`, the second test uses the connection string from the first test's `beforeAll`.

**Why it happens:** Node module cache keeps the singleton alive across tests in the same process.

**How to avoid:** For integration tests using mongodb-memory-server, set `process.env.MONGODB_CONNECTION_STRING` before the first import. Use `vi.resetModules()` before each test group that needs a fresh connection. Alternatively, export a `resetConnection()` function from `auth.ts` for test use.

**Warning signs:** Second test group gets `MongoServerError: connect ECONNREFUSED` or stale data from a previous test's database.

### Pitfall 2: `_id: 'lockout'` Type Cast

**What goes wrong:** `auth.ts` uses `_id: 'lockout' as unknown as import('mongodb').ObjectId` — a cast hack to satisfy TypeScript while using a string `_id`. Tests must use the same cast pattern when querying the auth collection directly.

**Why it happens:** MongoDB's TypeScript types default `_id` to `ObjectId`, but the code uses a string sentinel.

**How to avoid:** In integration tests, query as `collection.findOne({ _id: 'lockout' as unknown as ObjectId })`.

### Pitfall 3: ESM/CJS Module Mismatch in API Tests

**What goes wrong:** `api/tsconfig.json` uses `"module": "Node16"` which requires `.js` extensions on imports. Vitest may fail when mocking `mongodb` if the module resolution differs from the compiled output.

**Why it happens:** The API is compiled TypeScript targeting CommonJS-compatible Node16 modules. Vitest runs TypeScript directly but must match the resolution strategy.

**How to avoid:** Set `resolve.conditions: ['node']` in `api/vitest.config.ts`. Mock at the module ID level: `vi.mock('mongodb', ...)`.

### Pitfall 4: E2E Tests Need APP_PASSWORD in Environment

**What goes wrong:** Playwright starts `npx func start` which reads `api/local.settings.json`. But `local.settings.json` is gitignored — CI won't have it.

**Why it happens:** Azure Functions Core Tools reads `local.settings.json` for dev environment variables. In CI there's no such file.

**How to avoid:** Set `APP_PASSWORD` and `MONGODB_CONNECTION_STRING` as GitHub Actions environment variables (`env:` on the step or as Actions secrets). The Functions runtime reads them from the process environment directly.

### Pitfall 5: Playwright WebServer Race with MongoDB Startup

**What goes wrong:** `func start` connects to MongoDB on startup; if MongoDB isn't ready, Functions fail to start and Playwright times out.

**Why it happens:** The `supercharge/mongodb-github-action` step starts the container but may not wait long enough for MongoDB to accept connections.

**How to avoid:** Use the health-check built into the action (it waits for MongoDB port). Add a `timeout: 120_000` to the webServer that starts `func start`. Also set `reuseExistingServer: false` in CI so stale servers don't cause false passes.

### Pitfall 6: coverage-badges-action Needs Push Permission on Protected Branch

**What goes wrong:** Badge commits fail because `master` is branch-protected.

**Why it happens:** The action tries to push badge SVGs directly to the branch that triggered the workflow.

**How to avoid:** Configure the action to push badges to a separate unprotected branch (e.g., `badges`) or use the `no-commit: true` option and instead commit badges only on push to master (not PRs). The README badge URL can reference the `badges` branch.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| v3: mock context manually with `{} as Context` | v4: `new InvocationContext({...})` | v4 SDK (2023) | No stub library needed |
| v3: `function.json` bindings for test runner | v4: direct function import + `HttpRequest` constructor | v4 SDK (2023) | Tests are plain TypeScript modules |
| Jest as the Node test runner | Vitest (compatible API) | 2022 onwards | Already using Vitest in web; use same in API |
| Codecov for coverage badges | Self-hosted badges via `coverage-badges-action` | ongoing | No third-party account needed |

**Deprecated/outdated:**
- `stub-azure-function-context`: For v3 only. Do not use in v4.
- `runStubFunctionFromBindings`: v3 pattern that reads `function.json`. Incompatible with v4.
- `azure-functions-test-utils` (`@anthonychu/azure-functions-test-utils`): Provides `TestHttpRequest` and `TestContext`, but these are no longer needed since v4 classes are directly constructable. The library's `FuncCli` class (for starting Core Tools) could still be useful for integration-style tests, but Playwright's `webServer` is a cleaner alternative.

## Open Questions

1. **Should API Vitest use the same version as web (3.2.4) or install latest (4.1.0)?**
   - What we know: Both should work; web is pinned to `^3.2.4`.
   - What's unclear: Whether mixing vitest major versions across packages causes workspace confusion.
   - Recommendation: Match web's version (`^3.2.4`) in `api/` for consistency.

2. **Should lockout integration tests use module-level singleton reset or separate test file?**
   - What we know: `auth.ts` caches `MongoClient` at module scope; `vi.resetModules()` can force reimport.
   - What's unclear: Whether `vi.resetModules()` is reliable with dynamic imports in Vitest.
   - Recommendation: Export a `_resetConnectionForTest()` function from `auth.ts` (guarded by `process.env.NODE_ENV === 'test'`) to reset the singleton between tests.

3. **E2E: Preview vs Dev server for Vite?**
   - What we know: `vite preview` serves the built `dist/`; `vite dev` serves with HMR. Preview is closer to production behavior.
   - What's unclear: Build time in CI may slow E2E startup significantly.
   - Recommendation: Use `vite dev` with `reuseExistingServer: false` in CI for faster startup; switch to `vite preview` if flakiness occurs.
