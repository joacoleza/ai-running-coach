# Phase 1.1: Replace Auth with Simple Password - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace Azure SWA GitHub OAuth + owner role with a simple pre-shared password. Frontend shows a password prompt, stores the password in localStorage, and sends it as a header on every API call. The API checks the password against an env var and enforces a global brute-force lockout via MongoDB. Logout clears localStorage only — no server-side session to invalidate.

Deliverables per roadmap:
- Remove `staticwebapp.config.json` owner role lockdown and GitHub OAuth config
- `APP_PASSWORD` env var checked by API middleware on every protected request
- Frontend password prompt page (no GitHub login redirect)
- Password stored in localStorage; sent as custom header on every API call
- Logout clears localStorage only — no redirects to GitHub or Azure
- All existing routes protected behind password check

</domain>

<decisions>
## Implementation Decisions

### Frontend auth gate
- **D-01:** A `<PasswordPage />` component gates the entire app — rendered when `localStorage["app_password"]` is absent
- **D-02:** On correct password: store in `localStorage["app_password"]`, navigate to `/`
- **D-03:** On API 401 response: clear `localStorage["app_password"]`, redirect to password page
- **D-04:** Logout: delete `localStorage["app_password"]`, navigate to password page — no server call needed

### API auth middleware
- **D-05:** Custom header name: `X-App-Password`
- **D-06:** Every protected Azure Function reads `req.headers["x-app-password"]` and compares to `process.env.APP_PASSWORD`
- **D-07:** Health endpoint (`/api/health`) remains exempt from password check — but still returns 503 if site is globally locked
- **D-08:** Auth logic extracted into a shared middleware/helper so all functions call one place

### Brute-force lockout
- **D-09:** Global lockout state stored in MongoDB as a single document in a `auth` collection: `{ _id: "lockout", failureCount: N, blocked: boolean }`
- **D-10:** Every protected request checks `blocked` first — if `true`, return 503 immediately (before checking password)
- **D-11:** Wrong password → increment `failureCount` → if `failureCount >= 30`, set `blocked: true`
- **D-12:** Correct password → reset `failureCount` to 0 and `blocked` to false
- **D-13:** Lockout reset: manual — update the document via Azure Portal (Cosmos DB Data Explorer). No unlock endpoint needed.

### SWA config cleanup
- **D-14:** Remove owner role restriction from all routes — open `/*` and `/api/*` to `anonymous`
- **D-15:** Remove GitHub OAuth routes (`.auth/login/github`, `.auth/invitations/accept`, `.auth/logout`)
- **D-16:** Remove 401 redirect to `/.auth/login/github` — no longer applicable
- **D-17:** Keep `navigationFallback` rewrite to `index.html` — still needed for SPA routing
- **D-18:** Keep `platform.apiRuntime: node:22` — unchanged

### Claude's Discretion
- Password prompt visual design (form layout, button style)
- Error message copy on wrong password
- Whether to show failure count to user ("2 attempts remaining") or just a generic error
- MongoDB connection reuse pattern in Functions (module-level singleton)

</decisions>

<specifics>
## Specific Ideas

- "Anything simple and secure enough is good enough" — don't over-engineer the UI
- Brute-force protection is global (not per-IP) — 30 failures total locks the site. Reset via Azure Portal Data Explorer by updating the `auth` lockout document.
- Strong password is the real security control — the lockout is a backstop, not the primary defense

</specifics>
