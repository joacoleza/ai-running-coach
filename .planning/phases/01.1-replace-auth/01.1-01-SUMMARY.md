# Phase 1.1 Plan 01: Auth Middleware Summary

**MongoDB-backed password auth middleware with 30-failure global brute-force lockout, plus lockout-aware health endpoint**

## Performance

- **Duration:** 2 min
- **Started:** 2026-03-22T19:08:48Z
- **Completed:** 2026-03-22T19:11:13Z
- **Tasks:** 2
- **Files modified:** 2 (+ local.settings.json gitignored)

## Accomplishments

- Created `api/src/middleware/auth.ts` with `requirePassword` and `checkBlocked` exports
- `requirePassword` checks `X-App-Password` header, increments MongoDB failure counter on wrong password, locks site after 30 failures, resets on correct password
- `checkBlocked` reads MongoDB `auth.lockout` document — returns true if `blocked === true`
- Updated health endpoint to return 503 with `status: 'locked'` when globally locked out, preserving existing 200 `status: 'ok'` behavior

## Task Commits

Each task was committed atomically:

1. **Task 1: Create shared auth middleware with MongoDB lockout** - `8954b00` (feat)
2. **Task 2: Update health endpoint to check lockout state** - `840dd8c` (feat)

## Files Created/Modified

- `api/src/middleware/auth.ts` - Shared auth middleware: requirePassword + checkBlocked + MongoDB singleton
- `api/src/functions/health.ts` - Updated to import checkBlocked and return 503 when locked

## Decisions Made

- `local.settings.json` is gitignored by the project (established in Phase 1) so it was not committed. Developers must manually add `APP_PASSWORD` and `MONGODB_CONNECTION_STRING` to their local copy.
- The MongoDB lockout document uses `_id: 'lockout'` cast via `as unknown as ObjectId` to satisfy TypeScript strict mode — at runtime MongoDB accepts string `_id` values without issue.
- `requirePassword` returns `HttpResponseInit | null` (null = auth passed) so callers can write the idiomatic one-liner: `const authError = await requirePassword(req); if (authError) return authError;`
