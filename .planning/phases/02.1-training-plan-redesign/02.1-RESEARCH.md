# Phase 2.1: Training Plan Redesign - Research

**Researched:** 2026-03-25
**Domain:** React frontend data model migration, MongoDB schema change, SSE streaming XML tag protocol, HTML scraping for ChatGPT URL import
**Confidence:** HIGH

## Summary

This phase replaces the existing flat-sessions calendar plan with a hierarchical phases→weeks→days data model, a markdown-rendered plan view, inline editing, `<plan:update>` XML tag handling, ChatGPT URL import, and an archive section. All of the underlying infrastructure is already established: SSE streaming, XML tag stripping, ReactMarkdown, MongoDB plans collection, `requirePassword` middleware, and the `useChat` hook XML-command pattern.

The primary complexity is threefold: (1) migrating the MongoDB document schema from `sessions[]` to `phases[]` without breaking ongoing onboarding flows, (2) implementing the ChatGPT URL import on the backend via a direct HTTP GET + Claude parse, and (3) designing the `<plan:update>` tag schema clearly enough that Claude emits it correctly and the frontend can apply it surgically to the stored JSON.

The project uses no external component library — all UI is Tailwind + React. `react-markdown` is already installed and used in `ChatMessage.tsx`. The `react-big-calendar` dependency can be fully removed.

**Primary recommendation:** Work in five well-separated areas: types/schema, backend API endpoints, system prompt, frontend plan view + inline editing, and archive route. Keep each area independently testable.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Plan data model**
- D-01: Plan structure is hierarchical JSON: `phases[] → weeks[] → days[]` (not a flat sessions list, not a markdown blob)
- D-02: Phase fields: `name` (string, e.g. "Base Building"), `description` (string)
- D-03: Week fields: `weekNumber` (int), `startDate` (ISO date string)
- D-04: Day fields: `date` (ISO date string), `type` (`run` | `rest` | `cross-train`), `objective` (`{ kind: 'distance' | 'time', value: number, unit: 'km' | 'min' }` — optional on rest days), `guidelines` (string), `completed` (bool, default false), `skipped` (bool, default false)
- D-05: Plan-level fields: `objective` (enum: `marathon` | `half-marathon` | `15km` | `10km` | `5km`), `targetDate` (ISO date string), `status` (`onboarding` | `active` | `archived`)
- D-06: AI defines phase names and structure (base/build/peak/taper pattern) — user does not configure phases manually

**Plan view — display**
- D-07: Remove `react-big-calendar` entirely — no calendar widget
- D-08: Plan displayed as rendered markdown, computed from structured JSON (not stored as markdown)
- D-09: Rendering format: one section per phase, each phase shows weeks, each week shows its run days with objective and guidelines
- D-10: Completed days shown with visual indicator (strikethrough or checkmark); skipped days shown differently

**Plan view — actions**
- D-11: Training plan section has four actions: (1) Create new plan, (2) Import from ChatGPT URL, (3) Update plan, (4) Close & archive plan
- D-12: Every action the agent can perform is also directly available in the UI — no agent-only operations

**Editing — manual**
- D-13: Inline click-to-edit on each day's values (objective value, guidelines text). Click a field → becomes an input → save on blur/enter
- D-14: Mark day as completed or skipped via a toggle/button directly on the day row (no chat required)
- D-15: Editing is only available on days that are not yet completed — completed days are read-only

**Editing — via agent**
- D-16: Agent emits a `<plan:update>` XML tag to modify specific days or restructure phases. Frontend parses and applies it to the stored structured JSON
- D-17: Agent can only update non-completed days (same rule as manual editing)
- D-18: Full plan regeneration is possible via "Create new plan" after archiving — agent cannot silently replace an active plan

**ChatGPT URL import**
- D-19: User pastes a `chatgpt.com/share/...` URL into the import field
- D-20: Backend performs a direct HTTP GET to the URL, extracts readable text from the HTML response (no headless browser)
- D-21: Extracted text is sent to Claude with a prompt instructing it to parse and normalize into the `phases[]` plan structure
- D-22: On import success, plan is saved as active plan; existing active plan is archived first if one exists

**Archive**
- D-23: New "Archive" entry in the sidebar navigation
- D-24: Archive list shows: plan objective (e.g. "5K") and completion status — no date range
- D-25: Clicking an archived plan shows a readonly markdown view using the same rendering as the active plan
- D-26: "Close & archive" action on the active plan moves it to `status: archived` and returns user to the empty plan state

### Claude's Discretion
- Exact markdown rendering format for each phase/week/day (heading levels, table vs. list)
- Visual treatment for completed vs. skipped vs. upcoming days
- `<plan:update>` tag schema (what fields it carries to specify the target day and the new values)
- How to handle partial or ambiguous ChatGPT HTML extraction failures (show error vs. fallback to paste)

### Deferred Ideas (OUT OF SCOPE)
- Marking days complete via Apple Health run upload — Phase 3
- Date range display in archive list — future iteration
- Export plan to calendar format — noted in roadmap as v1.2
- Per-phase progress % in archive list — future iteration
- Editing completed-day notes/reflections after the fact — future iteration

## Architecture Patterns

### Type Schema (api/src/shared/types.ts)

Replace `PlanSession` and update `Plan`:

```typescript
// Source: CONTEXT.md D-01 through D-06
export interface PlanDay {
  date: string;           // ISO YYYY-MM-DD
  type: 'run' | 'rest' | 'cross-train';
  objective?: {
    kind: 'distance' | 'time';
    value: number;
    unit: 'km' | 'min';
  };
  guidelines: string;
  completed: boolean;
  skipped: boolean;
}

export interface PlanWeek {
  weekNumber: number;
  startDate: string;      // ISO YYYY-MM-DD
  days: PlanDay[];
}

export interface PlanPhase {
  name: string;           // e.g. "Base Building"
  description: string;
  weeks: PlanWeek[];
}

export interface Plan {
  _id?: ObjectId;
  status: 'onboarding' | 'active' | 'archived';
  onboardingMode: 'conversational' | 'paste';
  onboardingStep: number;
  summary?: string;
  goal: PlanGoal;
  objective?: 'marathon' | 'half-marathon' | '15km' | '10km' | '5km';

  createdAt: Date;
  updatedAt: Date;

```

**Note:** `sessions[]` is gone. `phases[]` replaces it. Existing active plans in MongoDB will have `sessions` but no `phases` — the GET /api/plan handler should treat a plan with no `phases` as needing re-generation (serve it as `onboarding` status or null).

### MongoDB Migration Strategy

The existing `plans` collection has documents with `sessions[]`. This phase introduces `phases[]`. Since this is a single-user personal tool:
- Any active plan with `sessions[]` but no `phases[]` is treated as stale — `GET /api/plan` returns null for such plans (or the plan page shows a "re-generate" prompt)
- No automated migration script is needed; the user can create a new plan
- The `discarded` status from the old schema can remain in MongoDB but will never be returned by `GET /api/plan`

### New API Endpoints


### Pattern: `<plan:update>` XML Tag Schema

The agent emits this tag to update non-completed days. The schema must be explicit in the system prompt so Claude gets it right:

```xml
<!-- Update objective and/or guidelines for a specific day -->
<plan:update date="2026-05-01" guidelines="Easy Zone 2 run — 45 min" />
<plan:update date="2026-05-01" objective_kind="distance" objective_value="8" objective_unit="km" />
<plan:update date="2026-05-01" completed="true" />
<plan:update date="2026-05-01" skipped="true" />
```

Design rationale:
- `date` is the primary key for locating a day across all phases/weeks
- Flat attributes (no nested JSON) — avoids XML attribute parsing complexity
- Each `<plan:update>` targets exactly one day
- Multiple `<plan:update>` tags in one response are allowed (e.g., restructure multiple days)
- Frontend parses all `<plan:update .../>` tags with a regex after stream completes, just like `<app:*/>` tags

Frontend parsing pattern (mirrors existing `<app:*/>` approach in `useChat.ts`):
```typescript
const planUpdateRegex = /<plan:update\s+([^/]+)\/>/g;
const updates = [...fullText.matchAll(planUpdateRegex)];
```

Parse attributes from each match using a simple attribute regex:
```typescript
function parseXmlAttrs(attrString: string): Record<string, string> {
  const result: Record<string, string> = {};
  const attrRegex = /(\w+)="([^"]*)"/g;
  for (const m of attrString.matchAll(attrRegex)) {
    result[m[1]] = m[2];
  }
  return result;
}
```

### Pattern: Markdown Rendering from Structured JSON

Compute markdown string on the frontend from the `Plan.phases[]` structure. Do NOT store markdown in MongoDB.

```typescript
// web/src/utils/planToMarkdown.ts
export function planToMarkdown(plan: PlanData): string {
  const lines: string[] = [];
  for (const phase of plan.phases) {
    lines.push(`## ${phase.name}`);
    if (phase.description) lines.push(`_${phase.description}_`);
    lines.push('');
    for (const week of phase.weeks) {
      lines.push(`### Week ${week.weekNumber}`);
      for (const day of week.days) {
        const prefix = day.completed ? '~~' : day.skipped ? '~~' : '';
        const suffix = day.completed ? '~~ ✓' : day.skipped ? '~~ (skipped)' : '';
        const objStr = day.objective
          ? `${day.objective.value}${day.objective.unit}`
          : day.type === 'rest' ? 'Rest' : '';
        lines.push(`- **${day.date}** ${prefix}${objStr} — ${day.guidelines}${suffix}`);
      }
      lines.push('');
    }
  }
  return lines.join('\n');
}
```

This function lives in `web/src/utils/planToMarkdown.ts` and is used by both the active plan view and the archived plan readonly view.

### Pattern: ChatGPT URL Import (D-19 to D-22)

Backend endpoint `POST /api/plan/import`:
1. Validates URL matches `^https://chatgpt\.com/share/`
2. Performs `fetch(url)` — Node.js built-in global in Azure Functions v4 (Node 18+)
3. Extracts text: strip HTML tags with a regex/simple parser (no `cheerio` needed for readable text)
4. Sends extracted text to Claude with a system prompt requesting `<training_plan_hierarchical>` JSON output
5. Parses response, validates structure, saves as active plan (archives existing active plan first)

**HTML text extraction (no headless browser):**
```typescript
function extractTextFromHtml(html: string): string {
  // Remove script/style blocks first
  const noScript = html.replace(/<script[\s\S]*?<\/script>/gi, ' ');
  const noStyle = noScript.replace(/<style[\s\S]*?<\/style>/gi, ' ');
  // Strip remaining tags
  const text = noStyle.replace(/<[^>]+>/g, ' ');
  // Collapse whitespace
  return text.replace(/\s+/g, ' ').trim();
}
```

**Error handling for import failures (Claude's Discretion):**
- URL fetch fails (404, CORS-equivalent network error) → return `{ error: 'Could not fetch the URL. Make sure the ChatGPT share link is public.' }`
- HTML too short / no training content detected → return `{ error: 'No training plan content detected in the shared conversation.' }`
- Claude fails to produce valid JSON → return `{ error: 'Could not parse a training plan from the conversation. Try pasting the content directly.' }`
- Frontend shows the error string in a banner; no fallback UI needed

### Pattern: Archive Flow (D-23 to D-26)

`POST /api/plan/archive` sets `{ status: 'archived', updatedAt: now }` on the current active plan.

`GET /api/plans/archived` queries `{ status: 'archived' }`, returns array of `{ _id, objective, goal, createdAt }`.

### Anti-Patterns to Avoid

- **Storing markdown in MongoDB:** Computed markdown breaks whenever the render logic changes; always derive from structured JSON (D-08)
- **Agent silently replacing plan:** D-18 locks this — the agent can only update days, not swap entire plans. Enforce this in the system prompt explicitly.
- **Editing completed days:** D-15/D-17 — both UI and PATCH endpoint must reject edits to days where `completed: true`
- **Using cheerio or jsdom for HTML extraction:** Brings large deps. A regex-based strip is sufficient for ChatGPT share pages (D-20)
- **Mixing `<plan:update>` stripping with `<training_plan>` stripping:** They are separate flows; keep them as separate regex passes in `useChat.ts`

## Common Pitfalls

### Pitfall 1: Schema Migration Breaks Active Plans
**What goes wrong:** Existing documents in MongoDB have `sessions[]` but no `phases[]`. After the code update, `plan.phases` is `undefined`, causing crashes in the plan view.
**Why it happens:** MongoDB is schema-less; old documents are not automatically migrated.
**How to avoid:** In `GET /api/plan`, after fetching, check: if `plan.sessions` exists and `plan.phases` is absent, return `null` (or treat plan as `status: onboarding` to trigger re-generation). Document this in STATE.md after execution.
**Warning signs:** `TypeError: Cannot read properties of undefined (reading 'map')` in the plan view.

### Pitfall 2: `<plan:update>` Stripped Before Stream Done
**What goes wrong:** If `<plan:update>` tags are stripped from the displayed message during streaming (before `done` event), partial tag text may flash in the UI.
**Why it happens:** The existing `<app:*/>` strip happens only on the `done` event. `<plan:update>` must follow the same pattern — strip only after stream is complete.
**How to avoid:** Add `<plan:update>` strip to the `payload.done` branch of `useChat.ts`, not to the streaming text update path.
**Warning signs:** Raw `<plan:update .../>` XML visible in the coach chat bubble.

### Pitfall 3: ChatGPT Share URLs May Require User-Agent Header
**What goes wrong:** Direct `fetch()` to `chatgpt.com/share/...` may return a 403 or a login redirect page rather than the shared content, because the server blocks bot requests.
**Why it happens:** CDN/server-side bot detection based on missing `User-Agent` or `Accept` headers.
**How to avoid:** Set realistic headers in the import fetch:
```typescript
const res = await fetch(url, {
  headers: {
    'User-Agent': 'Mozilla/5.0 (compatible; running-coach/1.0)',
    'Accept': 'text/html',
  },
});
```
If the response body is empty or contains a redirect-to-login pattern, return the appropriate error to the user. Mark this as a MEDIUM confidence concern — verify against the live URL in the CONTEXT.md specifics section.
**Warning signs:** HTTP 200 but extracted text contains "Log in" or is very short (< 500 chars).

### Pitfall 4: `date` Attribute Collision Across Phases
**What goes wrong:** Two phases could theoretically contain the same date (e.g., plan was restructured). `<plan:update date="X">` would match the first occurrence.
**Why it happens:** Date uniqueness is not enforced by the schema.
**How to avoid:** When generating a plan, the AI must produce non-overlapping date ranges. The system prompt should state: "Each date must appear at most once across all phases." The `PATCH /api/plan/days/:date` endpoint can use `findOneAndUpdate` with array filters to target the specific day without caring about phase position.
**Warning signs:** Wrong day being updated in a plan with multiple phases.

### Pitfall 5: `react-big-calendar` CSS Still Imported After Removal
**What goes wrong:** After `npm uninstall react-big-calendar`, there may still be an `import 'react-big-calendar/lib/css/react-big-calendar.css'` in `PlanCalendar.tsx`. If the file is deleted but the import remains elsewhere, the build will fail.
**Why it happens:** CSS imports are in `PlanCalendar.tsx` line 6. If the file is simply deleted, any remaining import will break Vite.
**How to avoid:** Delete `PlanCalendar.tsx` and `SessionModal.tsx` and remove any remaining imports. Also remove from `web/package.json` and run `npm install`.
**Warning signs:** Vite build error: `Cannot find module 'react-big-calendar/lib/css/react-big-calendar.css'`.

### Pitfall 6: `verbatimModuleSyntax` Type-Only Imports
**What goes wrong:** TypeScript tsconfig uses `verbatimModuleSyntax`. Importing types from shared modules without `import type` will cause a compilation error.
**Why it happens:** Established project pattern (noted in STATE.md for Phase 02-05 and 02-06).
**How to avoid:** All new files that import `PlanDay`, `PlanWeek`, `PlanPhase`, `Plan`, etc. purely as types must use `import type { ... }`.
**Warning signs:** `error TS1484: 'X' is a type and must be imported using a type-only import`.

### useChat.ts — `<plan:update>` Parsing (after stream done)

```typescript
// In the payload.done branch, after existing <app:*> handling:
const planUpdateRegex = /<plan:update\s+([^/]+)\/>/g;
const planUpdates = [...accumulatedText.matchAll(planUpdateRegex)];

if (planUpdates.length > 0) {
  // Strip from displayed message
  setMessages((prev) => {
    const updated = [...prev];
    const last = updated[updated.length - 1];
    if (last?.role === 'assistant') {
      updated[updated.length - 1] = {
        ...last,
        content: last.content.replace(planUpdateRegex, '').trim(),
      };
    }
    return updated;
  });

  // Apply each update via PATCH
  for (const match of planUpdates) {
    const attrs = parseXmlAttrs(match[1]);
    if (attrs.date) {
      await fetch(`/api/plan/days/${attrs.date}`, {
        method: 'PATCH',
        headers: authHeaders(),
        body: JSON.stringify(attrs),
      });
    }
  }

  // Re-fetch plan after all updates
  const updatedPlan = await fetchPlan();
  if (updatedPlan) setPlan(updatedPlan);
}
```

### planToMarkdown.ts — Discretionary Rendering Format

Recommended format (heading levels, visual indicators):
```typescript
export function planToMarkdown(plan: PlanData): string {
  const lines: string[] = [`# ${plan.objective ?? 'Training Plan'}`];
  for (const phase of plan.phases) {
    lines.push('');
    lines.push(`## ${phase.name}`);
    if (phase.description) lines.push(`*${phase.description}*`);
    for (const week of phase.weeks) {
      lines.push('');
      lines.push(`### Week ${week.weekNumber}`);
      for (const day of week.days) {
        if (day.type === 'rest') {
          lines.push(`- **${day.date}** — Rest`);
          continue;
        }
        const obj = day.objective
          ? `${day.objective.value} ${day.objective.unit}`
          : '';
        const status = day.completed ? ' ✓' : day.skipped ? ' _(skipped)_' : '';
        const text = `${obj}${obj && day.guidelines ? ' — ' : ''}${day.guidelines}`;
        // Strikethrough for completed/skipped
        const wrapped = (day.completed || day.skipped) ? `~~${text}~~` : text;
        lines.push(`- **${day.date}**${status} ${wrapped}`);
      }
    }
  }
  return lines.join('\n');
}
```

1. **Does `usePlan.ts` exist separately from `useChat.ts`?**
   - What we know: `PlanCalendar.tsx` imports from `../../hooks/usePlan`. That hook file was not found in the Glob results.
   - What's unclear: Whether `usePlan.ts` exists or was inlined. The Glob for `web/src/__tests__/PlanCalendar.test.tsx` suggests it may exist but not be in the main hooks directory.
   - Recommendation: Before deleting `PlanCalendar.tsx`, check for `web/src/hooks/usePlan.ts` and verify its usage. Delete it if it only served the calendar.

2. **ChatGPT share URL fetch reliability**
   - What we know: The example URL `https://chatgpt.com/share/695a515e-...` is a real user use case (CONTEXT.md). Direct `fetch()` may hit bot detection.
   - What's unclear: Whether chatgpt.com share pages are publicly accessible without JavaScript.
   - Recommendation: Implement with `User-Agent` header as shown above. Plan for the error path being exercised frequently. If fetch returns HTML with "Log in" text, surface a helpful error. This is a MEDIUM confidence concern — must be validated against the live URL during execution.

3. **`PlanGoal` vs plan-level `objective` / `targetDate`**
   - What we know: The existing `PlanGoal` type has `eventType`, `targetDate`, and profile info. D-05 adds plan-level `objective` and `targetDate` as top-level fields. There's overlap.
   - What's unclear: Whether `PlanGoal` should be kept (for onboarding context) or merged into the new plan-level fields.
   - Recommendation: Keep `PlanGoal` for backward compatibility (onboarding still uses it). Add `objective` and `targetDate` as additional top-level plan fields. The planner should resolve this explicitly in the plan.
