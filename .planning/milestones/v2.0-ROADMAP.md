# Milestone v2.0: Multi-User Support

**Status:** ✅ SHIPPED 2026-04-26
**Phases:** 6–10
**Total Plans:** 17

## Overview

Replaced the single-user APP_PASSWORD gate with full JWT-based multi-user authentication and an admin panel for managing user accounts. Each user now has isolated data (plans, runs, chat history). The system supports admin-provisioned accounts — no public registration.

## Milestone Summary

**Shipped:** 2026-04-26
**Timeline:** 2026-04-15 → 2026-04-26 (11 days)
**Stats:** 5 phases, 17 plans, ~248 files changed, 34,157 insertions / 2,398 deletions

### Key Accomplishments

1. Full JWT auth stack: login/refresh/logout endpoints with bcrypt password hashing, SHA-256 hashed refresh tokens, 15-min access tokens, 30-day refresh tokens
2. Complete frontend auth flow: LoginPage, ChangePasswordPage, App.tsx auth gate, global 401 interceptor with silent refresh + retry
3. Per-user data isolation: 48 userId-scoped MongoDB queries across 7 API handlers; startup migration backfills v1.1 orphaned documents to seed admin
4. Admin panel: list/create/reset/deactivate users with responsive mobile card layout; admin-provisioned accounts only; `active` flag enforced on every API request
5. IP-based login rate limiting: prevents credential stuffing and email enumeration; progressive lockout 15→24h; LoginPage handles 429 with API error body

### Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| JWT + bcrypt (not sessions) | Stateless auth fits Azure Functions; bcrypt is industry standard | ✓ Good — clean implementation |
| Server-side refresh token hash (SHA-256) | Never store raw refresh tokens | ✓ Good — secure by default |
| DB lookup on every requireAuth call | Instant deactivation without waiting for JWT expiry | ✓ Good — necessary for user management |
| /api/users prefix (not /api/admin/users) | Azure Functions Core Tools reserves /admin path prefix | ✓ Good — avoids 404 shadow |
| IP-based rate limiting (not per-user) | Prevents email enumeration; locks out attacker not victim | ✓ Good — cleaner security model |
| Admin-provisioned accounts only | Closed system; no signup friction; keeps Claude API costs low | ✓ Good — correct for small team |

### Issues Resolved

- Azure Functions Core Tools reserves /admin path — routed to /api/users instead
- ChangePasswordPage 401 loop in production — resolved by proactive token refresh before password change
- Rate limiting initially account-based (plan 10-01/02) → replaced with IP-based (plan 10-03) after discovering email enumeration vulnerability
