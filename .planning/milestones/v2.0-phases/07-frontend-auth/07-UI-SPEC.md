## Interaction Patterns

### Login Page

**States:**
1. **Initial state:** Email and password inputs empty, Log In button disabled (blue-300 background)
2. **Ready state:** Both fields have valid input, Log In button enabled (bg-blue-600 hover:bg-blue-700)
3. **Loading state:** Button text changes to "Checking...", button disabled (bg-blue-300), inputs disabled
4. **Error state:** Error message appears below form in red-600 text; inputs remain editable; button re-enables
5. **Success state:** Automatic redirect to /dashboard or /change-password (if tempPassword flag present)

**Form behavior:**
- Email input: required, any format accepted (server validates)
- Password input: required, type="password" (masked), no paste restrictions
- Form submit on Enter key (standard `<form onSubmit>`)
- Focus: autofocus email input on mount
- Button disabled when email OR password is empty
- On error, clear password field; preserve email for retry

**Focus states:**
- All inputs get `focus:outline-none focus:ring-2 focus:ring-blue-500 focus:border-blue-400` or similar
- Button gets standard keyboard focus ring (Tailwind default)

### Change Password Page

**States:**
1. **Initial state:** Both password inputs empty, Change Password button disabled
2. **Ready state:** Both fields have content AND passwords match AND length ≥8, button enabled
3. **Ready—mismatch state:** Both fields have content but don't match, button disabled, no error shown yet
4. **Error state:** Real-time validation feedback shown (e.g., "Passwords do not match" in red-600 below the confirm field) OR form-level error on submit attempt
5. **Loading state:** Button text "Saving...", button disabled, inputs disabled
6. **Success state:** Automatic redirect to /dashboard

**Form behavior:**
- New Password input: type="password" (masked), required
- Confirm Password input: type="password" (masked), required
- No "current password" field (user forced here, hasn't set own yet)
- Real-time feedback: show mismatch error only after user has typed both fields
- Minimum 8 characters enforced (no special char requirements)
- Form submit on Enter key
- On success, clear form and redirect (no toast/confirmation shown)
- On error, preserve entered values; show error message; re-enable button for retry

**Focus states:**
- Same as Login Page

### Logout Button (Sidebar)

**Behavior:**
- Click → `POST /api/auth/logout` with refreshToken
- Success → clear localStorage (access_token, refresh_token) + clear AuthContext state → auto-redirect to login
- Failure → still clear client state; auto-redirect to login (user is logged out locally)
- No confirmation dialog (per CLAUDE.md: window.confirm is used for destructive PLAN actions only, not general logout)
- Loading state: button text could change to "Logging out..." OR button becomes disabled during API call

**Styling:** Unchanged from v1.1 (gray-400 text, hover:bg-gray-800 hover:text-white, flex items-center layout, SVG icon + text)

## Accessibility Considerations

**Labels:** All form inputs have associated `<label>` with `htmlFor` attribute

**Form structure:**
- Inputs wrapped in `<form onSubmit={handleSubmit}>`
- Error messages announced via text rendering (not aria-live, sufficient for single-form context)

**Keyboard navigation:**
- Tab order: email → password → submit button (natural DOM order)
- Enter key submits form (standard behavior)
- Escape key: NOT used (no modal to close; page refresh or back button for exit)

**Color contrast:**
- Body text (gray-900) on light bg (gray-50/white): 16.3:1 ✓
- Error text (red-600) on white: 4.5:1 ✓
- Button text (white on blue-600): 8.6:1 ✓

## API Integration Points

**Login form:**
```
POST /api/auth/login
body: { email, password }
response: { token, refreshToken, expiresIn, tempPassword }
```

**Change Password form:**
```
POST /api/auth/change-password (or PATCH /api/auth/password — define in phase planning)
body: { newPassword }
response: 200 OK (or 204)
requires: valid JWT in Authorization header
```

**Refresh token (silent, not visible on page):**
```
POST /api/auth/refresh
body: { refreshToken }
response: { token, expiresIn }
```

**Logout (Sidebar button):**
```
POST /api/auth/logout
headers: Authorization: Bearer <token>
body: { refreshToken }
response: 204 No Content
```

## Router Integration (App.tsx)

**Gate logic (defined in CONTEXT.md D-03):**

```
if (!authenticated) {
  return <LoginPage />
}
if (authenticated && tempPassword) {
  return <ChangePasswordPage />
}
return <BrowserRouter>...</BrowserRouter>
```

**No URL routes for login/change-password:** These pages exist above BrowserRouter, not as Route entries.

**Silent 401 interceptor in App.tsx:**
- Catches 401 responses from fetch calls
- Calls `POST /api/auth/refresh` with current refreshToken
- If refresh succeeds → updates access_token in localStorage + retries original request
- If refresh fails (401) → clears auth state + redirects to login
- Does NOT intercept `/api/auth/login` or `/api/auth/refresh` calls themselves
